# Mnemo v0.3.0 benchmark ŌĆö 2026-04-21

**Commit under test:** `dd8c6718` (main after v0.3.0 merge).
**Harness:** `python -m mnemo.benches.locomo_runner`.
**Hardware:** Apple Silicon, macOS, DuckDB embedded backend, `USearch` HNSW
vector index, `NoopEmbedding` provider (no `OPENAI_API_KEY` set ŌĆö see
"Known gaps" below).
**Datasets:**
- **LoCoMo-MC10** ŌĆö `Percena/locomo-mc10`, `train` split, first 20 rows
  (CC BY-NC 4.0; attribution: Maharana et al., "Evaluating Very Long-Term
  Conversational Memory of LLM Agents", arXiv:2402.17753).
- **LongMemEval-cleaned** ŌĆö `xiaowu0162/longmemeval-cleaned`, split
  `longmemeval_oracle`, first 20 rows (Wu et al., ICLR 2025). The
  `longmemeval_s_cleaned` variant removes the answer-bearing sessions from
  the haystack; we use `longmemeval_oracle` because the task here is
  retrieval, not noise-robust inference.

**Latency budget:** every strategy lands well under the v0.3.0 target of

zero, for two root causes identified below.**

## Why recall is zero ŌĆö root-cause findings

These numbers are not a benchmark of Mnemo's retrieval quality under
realistic operating conditions. They are a benchmark of Mnemo's retrieval
quality **with two v0.3.0 bugs the run exposed**:

1. **The Python `MnemoClient` does not attach a full-text index.**
   `python/src/lib.rs` constructs a bare `MnemoEngine` without
   `.with_full_text(...)`, so the `lexical` and `hybrid_rrf` strategies
   have no BM25 signal when driven from Python. Rust-side callers that
   build the engine themselves are unaffected. Tracked ŌĆö see v0.3.1
   roadmap.

2. **`NoopEmbedding` is the default when `OPENAI_API_KEY` is unset.** The
   noop provider returns identical vectors for every input, which collapses
   semantic similarity to noise. The `vector_only`/`auto`/`graph_boosted`
   strategies inherit that, producing essentially random retrieval from
   the ~10 000-memory pool. Fix: set `OPENAI_API_KEY`, or wire local ONNX
   embeddings (currently broken against the shipped `ort` API ŌĆö see
   v0.3.0 CHANGELOG "Deferred" section).

**Implication:** these are the floor numbers. Any fix to either bug will
lift them substantially. The v0.3.2 LLM-as-judge scorer will also raise
LoCoMo's number against inferential answers ("MBA" ŌåÆ "Business
Administration"-style), where substring match is a wrong default.

## Reproduce

```bash
# Build the native extension once
cd python && maturin develop --release

# Pull the benchmark extra
pip install "mnemo[benchmark]"

# LoCoMo run
python -m mnemo.benches.locomo_runner \
    --dataset locomo \
    --max-examples 20 \
    --db-path /tmp/bench_locomo.mnemo.db \
    --strategies auto vector_only hybrid_rrf graph_boosted lexical

# LongMemEval run (oracle split)
python -m mnemo.benches.locomo_runner \
    --dataset longmemeval \
    --split longmemeval_oracle \
    --max-examples 20 \
    --db-path /tmp/bench_lme.mnemo.db \
    --strategies auto vector_only hybrid_rrf graph_boosted lexical
```

## Tracking issues opened from this run

- [ ] **PyO3 `MnemoClient` missing full-text index** ŌĆö wire Tantivy in the
  Python constructor so Python callers can run the `lexical`/`hybrid_rrf`
  paths. Same fix recommended for the admin surface.
- [ ] **LLM-as-judge scorer** ŌĆö substring scoring is too strict for
  LongMemEval-style inferential gold answers. Integrate a small judge
  (gpt-4o-mini or similar) with confidence thresholding; use it for
  LongMemEval, keep substring for LoCoMo-MC10's multiple-choice answers.
- [ ] **Repair `onnx` embedding feature** ŌĆö the v0.3.0 CHANGELOG already
  noted `ort` API drift; without local embeddings, benchmark runs that
  can't use OpenAI will always degrade to `NoopEmbedding`.
- [ ] **Extend harness with `--backend postgres`** ŌĆö Task 1 spec asked
  for a pgvector run; deferred pending a local Postgres + pgvector 0.8.2
  fixture.
